#### 资源处理流程

一开始需要指定一个/多个入口（entry）也就是告诉 webpack 从源码目录下的哪个文件开始打包。入口相当于一棵树的根.

- chunk: 存在依赖关系的模块会在打包的时候被封装为一个 chunk

webpack 会从入口文件开始检索，将具有依赖关系的模块生成一棵依赖树，最终会得到一个 chunk（一棵依赖树就是一个 chunk）。由这个 chunk 的到的产物我们称为一个 bundle

在工程中可以定义多个入口，每一个入口都会产生一个结果资源，因此可以说，entry 与 bundle 存在着对应关系。特殊情况下 一个入口也可以产生多个 chunk 并输出多个 bundle

#### 配置资源入口

由 context 和 entry 共同决定入口文件的路径，配置入口实际上做了两件事

1. 确定入口模块的位置，告诉 webpack 从哪个文件开始打包

2. 定义 chunk name， 如果只有一个入口 那么 chunk name 就是 main 如果有多个入口，我们需要为每一个入口定义 chunk name，作为该 chunk 的唯一标识

##### context

可以理解为资源入口的前缀，必须是**绝对路径**，只能是字符串

context 可以省略 默认是项目的根目录

入口文件 src/index/js

```
const path = require("path");
module.exports = {
  context: path.join(__dirname),
  entry: "./src/index.js",
  output: {
    path: path.resolve(__dirname, "dist"),
    filename: "bundle.js"
  },
  mode: "development"
};

// 等价于

const path = require("path");
module.exports = {
  context: path.join(__dirname,'./src'),
  entry: "./index.js",
  output: {
    path: path.resolve(__dirname, "dist"),
    filename: "bundle.js"
  },
  mode: "development"
};
```

##### entry

和 context 不同 entry 可以是字符串，对象，数组，函数

- 字符串

```
const path = require("path");
module.exports = {
  entry: "./src/index.js",
  output: {
    path: path.resolve(__dirname, "dist"),
    filename: "bundle.js"
  },
  mode: "development"
};
```

- 数组

传入一个数组的作用是将多个资源预先合并，在打包时 Webpack 会将数组中的最后一个元素作为实际的入口路径

```
const path = require("path");
module.exports = {
  entry: ["./src/index1.js", "./src/index.js"],
  output: {
    path: path.resolve(__dirname, "dist"),
    filename: "bundle.js"
  },
  mode: "development"
};
```

相当于

```
// webpack.config.js
const path = require("path");
module.exports = {
  entry: "./src/index.js",
  output: {
    path: path.resolve(__dirname, "dist"),
    filename: "bundle.js"
  },
  mode: "development"
};

// index.js
import './index1.js'
```

- 对象

如果想定义多个入口 就必须使用对象的形式 key 为 chunk name，value 就是模块路径

value 可以是字符串，也可以是数组

```
const path = require("path");
module.exports = {
  context: path.join(__dirname),
  entry: {
    index: "./src/index.js",
    lib: "./src/lib.js"
  },
  output: {
    path: path.resolve(__dirname, "dist"),
    filename: "bundle.js"
  },
  mode: "development"
};
```

- 函数

使用函数类型的入口时，只要返回值是上面任意一种类型就可以

```
const path = require("path");
module.exports = {
  context: path.join(__dirname),
  entry: () => ({
    index: "./src/index.js",
    lib: "./src/lib.js"
  }),
  output: {
    path: path.resolve(__dirname, "dist"),
    filename: "bundle.js"
  },
  mode: "development"
};
```

传一个函数的优点就是 我们可以在函数中传入一些动态逻辑来确定项目的入口

函数也支持返回一个 Promise 对象进行异步操作

```
const path = require("path");
module.exports = {
  context: path.join(__dirname),
  entry: () =>
    new Promise((resolve, reject) => {
      setTimeout(() => {
        resolve("./src/index/js");
      }, 1000);
    }),
  output: {
    path: path.resolve(__dirname, "dist"),
    filename: "bundle.js"
  },
  mode: "development"
};
```

#### 实例

##### 单页应用

优点： 入口文件只有一个，依赖关系清晰

缺点：所有模块都被打包到一起，当应用规模变大之后会导致打包后的文件过大，降低用户页面渲染速度，一旦代码有一点点改动，用户都需要重新下载这个很大的 bundle，性能很不友好

##### 提取 vender（供应商）

webpack 将一些第三方的模块进行单独打包到一个 bundle 中，不会经常变动，所以可以利用缓存

```
const path = require("path");
module.exports = {
  context: path.join(__dirname),
  entry: {
    index: "./src/index.js",
    vender: ["react", "react-dom"]
  },
  output: {
    path: path.resolve(__dirname, "dist"),
    filename: "bundle.js"
  },
  mode: "development"
};
```

#### 多页应用

```
const path = require("path");
module.exports = {
  context: path.join(__dirname),
  entry: {
    // 每个页面的 js 单独打包为一个 bundle 每个页面的html只需要引入自己的 bundle 即可
    pageA: "./src/pageA.js",
    pageB: "./src/pageB.js",
    pageC: "./src/pageC.js",
    vender: ["react", "react-dom"]
  },
  output: {
    path: path.resolve(__dirname, "dist"),
    filename: "bundle.js"
  },
  mode: "development"
};
```

#### 配置资源出口

基本配置

```
const path = require("path");
module.exports = {
  context: path.join(__dirname),
  entry: "./src/index.js",
  output: {
    path: path.resolve(__dirname, "dist"),
    // 配置打包后的文件的文件名
    // 不仅可以是一个文件名 还可以是一个相对路径 即使这个目录不存在也没有关系 webpack 会帮助我们创建 是相对于 dist 目录而言的
    filename: "bundle.js",
    publicPath: "/dist"
  },
  mode: "development"
};
```

- filename 选项

filename: '[name].bundle.js'

这个 name 是一个占位符 会被替换为 entry 配置的 chunk name

其他模版变量

    | 变量名称 | 功能描述 |
    | ----  | ---- |
    | hash | 哈希值，每次build之后的 hash 值都不同 |
    | chunkhash | 当前chunk内容的 hash 值 |
    | id | 当前chunk的id |
    | query | 指代filename配置项中的query |

模版变量的作用

1. 对于不同的 chunk 进行区分

2. 缓存

- path 指定输出资源的路径 是一个绝对路径

```
const path = require("path");
module.exports = {
  context: path.join(__dirname),
  entry: "./src/index.js",
  output: {
    path: path.resolve(__dirname, "dist"),
    filename: "bundle.js"
  },
  mode: "development"
};
```

- publicPath

path 用来指定资源的输出位置，publicPath 用来指定资源的请求位置

> 1. 输出位置：打包完成之后资源产生的目录，一般是根目录下的 dist 目录

> 2. 请求位置：由 JS 或者 CSS 请求的间接资源路径。页面中的资源分为两种，一种是 html 页面直接请求的（script 标签）；另一种是由 JS 或者 CSS 请求的，如异步加载的 JS 在 CSS 中请求的图片，字体等。publicPath 的作用就是指定这部分间接资源的请求位置。

publicPath 有三种形式

1. HTML 相关

我们可以将 publicPath 指定为相对路径，请求这些资源的时候就会使用当前 HTML 的路径再加上相对路径，构成实际请求的 url

```
// 假设当前HTML地址为 https://example.com/app/index.html
// 异步加载的资源名为 0.chunk.js
publicPath: "" // 实际路径https://example.com/app/0.chunk.js
publicPath: "./js" // 实际路径https://example.com/app/js/0.chunk.js
publicPath: "../assets/" // 实际路径https://example.com/aseets/0.chunk.js
```

2. HOST 相关

pathPath 以 / 开始，代表 publicPath 是以当前 hostname 为基础的

```
// 假设当前HTML地址为 https://example.com/app/index.html
// 异步加载的资源名为 0.chunk.js
publicPath: "/" // 实际路径https://example.com/0.chunk.js
publicPath: "/js/" // 实际路径https://example.com/js/0.chunk.js
publicPath: "/dist/" // 实际路径https://example.com/dist/0.chunk.js
```

3. CDN 相关

上面都是相对路径的配置，我们也可以使用绝对路径配置 publicPath
这种情况一般发生于静态资源放在 CDN 上面时，由于其域名与当前页面域名不一致，需要以绝对路径的形式进行指定。

当 publicPath 以协议名开始，则表示当前路径是与 CDN 相关的

```
// 假设当前页面路径为 https://example.com/app/index.html
// 异步加载的资源名为 0.chunk.js
publicPath: "http://cdn.com/" // 实际路径http://cdn.com/0.chunk.js
publicPath: "https://cdn.com/" // 实际路径https://cdn.com/0.chunk.js
publicPath: "//cdn.com/assets/" 实际路径 //cdn.com/assets/0.chunk.js
```

> webpack-dev-server 的配置中也有一个 publicPath; 但是这个 publicPath 是用来指定静态资源的加载路径

```
const path = require('path');
module.exports = {
  entry: './src/app.js',
  output: {
    filename: 'bundle.js',
    path: path.join(__dirname, 'dist'),
  },
  devServer: {
    publicPath: '/assets/',
    port: 3000,
  },
};
```

但是当我们启动 webpack-dev-server 的服务后，访问 localhost：3000/dist/bundle.js 时却会得到 404。这是因为 devServer.publicPath 配置项将资源位置指向了 localhost：3000/assets/，因此只有访问 localhost：3000/assets/bundle.js 才能得到我们想要的结果。

为了避免开发环境和生产环境 publicPath 不同而导致的疑惑 我们可以让 devServer.public 和 output.path 保持一致，这样在任何环境下资源的输出路径都是一样的

#### 小结

当第三方依赖比较多的时候 我们可以使用提取 vender 的方法将一些第三方模块提取到单独的 chunk，有效利用客户端缓存，加快页面渲染速度
