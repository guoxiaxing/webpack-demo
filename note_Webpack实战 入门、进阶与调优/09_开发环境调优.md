#### 模块热替换 HMR 开发环境下使用

- live reload 监测到代码改变就自动构建，刷新浏览器

- HMR 在页面不刷新的情况下得到最新改动

1. 开启 hmr

首先我们要确保项目是基于 webpack-dev-server 或者 webpack-dev-middle 进行开发的，Webpack 本身的命令行并不支持 HMR。

```
const webpack = require('webpack');
module.exports = {
  // ...
  plugins: [
    new webpack.HotModuleReplacementPlugin()
  ],
  devServer: {
    hot: true,
  },
};
```

```
// index.js
import { add } from 'util.js';
add(2, 3);

if (module.hot) {
  module.hot.accept();
}
```

假设 index.js 是应用的入口，那么我们就可以把调用 HMR API 的代码放在该入口中，这样 HMR 对于 index.js 和其依赖的所有模块都会生效。当发现有模块发生变动时，HMR 会使应用在当前浏览器环境下重新执行一遍 index.js（包括其依赖）的内容，但是页面本身不会刷新。

2. 原理

在开启 hmr 之后会发现打包后的资源体积比之前大了许多，这是因为 webpack 为了实现 hmr 注入了一些代码

开发环境下 浏览器是客户端 webpack-dev-server 是服务器

hmr 的核心就是从服务器端拉取更新后的资源（准确的说，hmr 拉取的不是整个资源，而是进行 chunk diff，只拉取更新的部分）

- 浏览器什么时候去拉取这些更新

需要 webpack-dev-server 对本地文件进行监听

实际上 WDS 与浏览器之间维护了一个 websocket，当本地资源发生变化时 WDS 会向浏览器推送更新事件，并带上这次构建的 hash，让客户端与上一次资源进行比对。通过 hash 的比对可以防止冗余更新的出现。因为很多时候源文件的更改并不一定代表构建结果的更改（如添加了一个文件末尾空行等）。

当然 webscoket 并不是只有开启了 HMR 才会有，live reload 其实也是依赖这个而实现的。

- 拉取什么

客户端已经知道新的构建结果和当前的有了差别，就会向 WDS 发起一个请求来获取更改文件的列表，即哪些模块有了改动。通常这个请求的名字为[hash].hot-update.json。

#### HMR

- 代码变更后通过 websocket 将变更的部分代码替换到页面中，并无需刷新浏览器直接更新视图

- webpack-dev-server 维护一个与浏览器间的 websocket 连接

- webpack-dev-server 监控到本地文件变化，将本次构建 hash 推送到浏览器

- 浏览器与上一次资源进行比对，判断是否需要拉取更新，若需要则发送请求获取更改文件列表

请求：[hash].hot-update.json；

返回值：hash 版本+chunk name；

- websocket 返回结果告知需要更新的 chunk name 和 版本 hash

- 浏览器使用获得的 chunk name 和 hash 请求增量更新

- 将获取到的更新，通过一定代码，进行视图热更新
